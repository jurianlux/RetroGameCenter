# アーキテクチャの決定事項

> 重要な設計判断とその理由を記録する。変更時は理由とトレードオフを必ず記載する。

## 決定事項ログ

### [ARCH-001] Python + Pygame での実装

- **日付**: 2026-06-27
- **決定**: Python を言語、Pygame をゲームフレームワークとして採用
- **理由**: Pygame は 2D レトロゲーム開発に適切で、学習曲線が緩やか。ドット絵・タイルマップ・サウンド処理も標準機能で対応可能
- **却下した代替案**: Unity（重い）、Godot（3D 寄り）、JavaScript + Canvas（Web 環境が不要）
- **影響範囲**: 開発環境、ライブラリ選定、デプロイ方法

### [ARCH-002] Tiled タイルマップエディタの採用

- **日付**: 2026-06-27
- **決定**: ステージデザインに Tiled エディタを使用、pytmx で読み込み
- **理由**: ビジュアルエディタで直感的にステージを設計でき、複数ステージの管理が容易。ドンキーコング以降のゲーム追加でも再利用可能
- **却下した代替案**: ハードコーディング（スケーラビリティ不足）、CSV ファイル（可視性が低い）
- **影響範囲**: マップ設計フロー、ステージ読み込み実装

### [ARCH-003] スコア記録・ランキング機能

- **日付**: 2026-06-27
- **決定**: JSON ファイルでスコアを記録し、メニュー画面でランキング表示
- **理由**: シンプルで実装が容易。Python の json モジュール標準機能で扱える。ローカルデスクトップアプリなので十分
- **却下した代替案**: SQLite（複雑さが不要）、サーバー保存（マルチプレイなし）
- **影響範囲**: ゲーム終了時の処理、メニュー画面表示

### [ARCH-004] サウンド機能

- **日付**: 2026-06-27
- **決定**: pygame.mixer で BGM・SE を再生
- **理由**: Pygame 標準機能。WAV・MP3・OGG 等の形式に対応。追加ライブラリ不要
- **影響範囲**: アセット管理（音声ファイル）、各ゲームシーンの実装
